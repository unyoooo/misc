🔤 テキスト エンコード / デコード ツール

テキストを入力すると、URLエンコード・Base64・Hex（バイト列）・Unicodeコードポイントなどへ一括変換します。 UTF-8 と Shift_JIS の両方のバイト表現に対応（変換表はブラウザ内で生成・送信なし）。 下段では逆方向のデコードと、文字化けの再現もできます。

📝 エンコード（テキスト → 各形式）

文字数 - UTF-16要素 - UTF-8 - bytes Shift_JIS - bytes
URLエンコード（UTF-8）
URLには日本語・スペース・記号をそのまま書けないため、%XX形式に変換して埋め込む。検索クエリやAPIのパラメータ（?q=%E3%81%82）で毎日使われている、いまのWebの標準形式。JavaScriptでは encodeURIComponent()。
URLエンコード（Shift_JIS）
昔のWebサイトやCGI（掲示板・検索フォーム）はShift_JISで動いていたため、URLエンコードもSJISバイト基準だった。レガシーなシステムへのリクエストの再現・デバッグ用で、新規に使うことはまず無い。同じ「あ」でも %E3%81%82（UTF-8）と %82%A0（SJIS）で別物になるのがポイント。
Base64（UTF-8）
どんなバイト列も英数字と+/=だけの文字列にする仕組み。「テキストしか通せない場所」にデータを通すために使う：メールの添付ファイル（MIME）、画像をHTML/CSSに直接埋め込むデータURI（data:image/png;base64,…）、Basic認証、JSONへのバイナリ埋め込みなど。⚠ 暗号化ではない（誰でも元に戻せる）ので秘密情報の保護には使えない。
Base64（Shift_JIS）
中身のバイト列がShift_JISになっているBase64。古いメールシステムやレガシーな業務システムとのデータ交換で登場する。「Base64をデコードしたのに化ける」ときは、中身がUTF-8ではなくSJISだったというパターンの可能性が高く、その確認に使える。
Hex（UTF-8バイト）
1バイトを16進数2桁で表した、「実際に並んでいるバイトそのもの」。バイナリエディタ・デバッガ・通信ログ（シリアルモニタ等）に出てくるのがこの形式で、文字化け調査の最終確認に使う。プログラムでバイト列を直接書くとき（0xE3, 0x81, 0x82）にも。先頭が EF BB BF ならBOM付きUTF-8。
Hex（Shift_JISバイト）
Shift_JISでのバイト列。組み込み機器・レシートプリンタ・古い計測器・LCDモジュールなどは今でもSJISでテキストを受け取ることが多く、マイコンからシリアル通信で送るバイト列の確認・デバッグに使う。半角カナが1バイト（B1＝ｱ）で済むのもSJISの特徴。
Unicodeコードポイント
文字に1つずつ割り当てられた「世界共通の背番号」（あ＝U+3042）。見た目がそっくりな別文字（ー長音とU+2212マイナス、〜波ダッシュとU+FF5E全角チルダ等）を確実に区別できるので、「この文字の正体は何？」の調査に使う。プログラムで文字を指定するとき（'あ'）の基礎情報でもある。
HTML数値文字参照（&#x…;）
HTMLの中に文字を番号で書く記法（&#x3042; → あ）。ページの文字コードが何であっても壊れないので、SJISのページにSJISに無い文字（絵文字や特殊記号）を入れたいときや、©・→ などの記号を確実に表示したいときに使う。

🔍 文字化けの見分け方チートシート

「縺ゅ→縺」「繧繝」系
UTF-8のデータをShift_JISとして読んでいる。読み込み側の文字コード設定をUTF-8に。
「ã�‚ã�„」「ã‚ã„ã†」系
UTF-8のデータをLatin-1(Windows-1252)として読んでいる。海外製ツールやメールでよく起こる。
「�����」（菱形だらけ）
Shift_JISなどのデータをUTF-8として読んでいる。UTF-8として不正なバイトが「�」に置換された状態。
「?????」（ハテナだらけ）
変換時に文字が失われている。変換先の文字コードに無い文字（絵文字をShift_JISへ等）が「?」に置換された。読み方を変えても戻らない。
先頭に「ï»¿」や「﻿」
BOM（EF BB BF）。UTF-8ファイルの先頭印。プログラムでCSV等を読むとき邪魔になることがある。
URLの「+」がスペースになる
フォーム形式（application/x-www-form-urlencoded）ではスペースを「+」で表す流儀がある。このツールのデコードは「+」をスペースとして扱う。
変換はすべてブラウザ内で完結します（どこにも送信されません）。Shift_JIS変換はブラウザのデコーダ（CP932相当）から逆引き表を生成しています。
